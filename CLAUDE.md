## 코딩 전 사고
- 가정하지 말고, 불확실하면 질문한다
- 해석이 여러 개 가능하면 선택지를 제시한다 (조용히 하나를 고르지 않는다)
- 더 단순한 접근이 있으면 말한다. 필요하면 반론을 제기한다

## Development Principles
- **TDD (서비스 로직, 비즈니스 규칙, 데이터 레이어, 유틸리티)**: 테스트 먼저 작성 → 실패 확인(Red) → 최소 구현(Green) → 리팩토링(Refactor)
- **TDD 선택 (UI 컴포넌트, 단순 CRUD 위임)**: 구현 후 테스트 작성도 허용
- **단순 설계**: 동작하게 만들고, 올바르게 만들고, 빠르게 만들어라 (Make it work, make it right, make it fast)
  - 요청받지 않은 기능, 추상화, 설정 가능성을 추가하지 않는다
  - 일어날 수 없는 시나리오에 대한 에러 처리를 넣지 않는다
  - 200줄이 50줄로 가능하면 다시 쓴다
- **작은 단계**: 한 번에 하나의 동작만 변경, 각 단계마다 테스트 통과 확인

## 최소 변경 원칙
- 변경한 모든 줄은 사용자의 요청과 직접 연결되어야 한다
- 인접한 코드, 주석, 포매팅을 "개선"하지 않는다
- 기존 스타일이 내 취향과 달라도 기존 스타일을 따른다
- 내 변경으로 생긴 미사용 import/변수/함수는 제거한다. 기존 dead code는 언급만 하고 삭제하지 않는다

## 목표 기반 실행
- 작업을 검증 가능한 목표로 변환한다:
  - "유효성 검사 추가" → "잘못된 입력에 대한 테스트 작성 후 통과시키기"
  - "버그 수정" → "재현 테스트 작성 후 통과시키기"

## Workflow Rules
- 같은 파일 3회 이상 편집 시: 멈추고 접근 방식을 재고한다
- "String not found" 2회 발생 시: 파일을 다시 Read한 후 재시도
- 테스트 3회 연속 실패 시: 현재 접근을 중단하고 대안을 제시한다
- 동일 에러 반복 시: 에러 메시지를 정확히 읽고 근본 원인을 분석한다
