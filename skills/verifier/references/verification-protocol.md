# 4단계 검증 프로토콜

## Phase 1: 요구사항 분해 (Requirement Decomposition)

확보한 기준을 **원자적(atomic) 검증 항목**으로 분해합니다.

### 원자적 항목의 조건

각 항목은 아래 조건을 모두 충족해야 합니다:

1. **하나의 기능만 다룬다** — 복합 항목은 분해
2. **검증 가능한 동사로 끝난다** — 존재한다, 동작한다, 반환한다, 포함한다, 저장된다
3. **코드에서 증거를 찾을 수 있다** — 파일, 함수, 테스트 등

### 분해 예시

```
원본: "SWL 보고서 쿼리 구현"

분해:
  1. SwlPipe 조회 쿼리가 존재한다
  2. 날짜 필터 파라미터가 동작한다
  3. 지역 필터 파라미터가 동작한다
  4. 페이지네이션이 동작한다
  5. 정렬 기준이 적용된다
  6. 응답 DTO에 필요한 필드가 포함된다
  7. 빈 결과 시 빈 리스트를 반환한다
  8. 잘못된 파라미터 시 400 에러를 반환한다
```

### 검증 불가능한 항목 처리

아래와 같은 항목은 검증 불가능합니다:

```
Bad:
  - "구현 완료"
  - "잘 동작함"
  - "처리됨"
  - "적절히 구현"

→ 이런 항목을 만나면 목표/요청에서 구체적 항목으로 재도출
```

## Phase 2: 증거 기반 대조 (Evidence-Based Verification)

각 검증 항목에 대해 **코드에서 증거를 찾고 판정**합니다.

### 증거 수집 절차

```
각 검증 항목에 대해:
  1. 관련 파일을 탐색한다 (Glob, Grep)
  2. 해당 파일을 읽는다 (Read)
  3. 항목이 충족되었는지 판단한다
  4. 판정과 증거를 기록한다
```

### 증거의 형태

| 증거 유형 | 예시 |
|----------|------|
| 코드 존재 | `UserService.kt:45` — register() 함수 존재 |
| 테스트 존재 | `UserServiceTest.kt:30` — 중복 이메일 테스트 |
| 설정 존재 | `application.yml:12` — Redis 설정 |
| API 엔드포인트 | `UserController.kt:20` — POST /v1/users 매핑 |
| 쿼리 존재 | `UserRepository.kt:15` — findByEmail() 정의 |

### 판정 규칙

**PASS** — 증거가 명확하고 항목을 완전히 충족
```
항목: "이메일 중복 검사가 동작한다"
증거: UserService.kt:52 — existsByEmail() 호출 후 예외 발생
판정: ✅ PASS
```

**FAIL** — 관련 코드가 존재하지 않거나 항목을 충족하지 않음
```
항목: "정렬 기준이 적용된다"
증거: SwlPipeRepository.kt — sortBy 파라미터 없음
판정: ❌ FAIL
```

**PARTIAL** — 일부만 구현되었거나 엣지 케이스가 누락됨
```
항목: "에러 케이스가 처리된다"
증거: UserController.kt:35 — 빈 결과 처리 있음, 잘못된 파라미터 처리 없음
판정: ⚠️ PARTIAL — 잘못된 파라미터 케이스 미처리
```

## Phase 3: 변경 영향 리뷰 (Change Impact Review)

작업으로 인한 **의도하지 않은 변경**이 없는지 확인합니다.

### 체크 항목

```
1. 범위 확인
   - git diff --stat으로 변경된 파일 목록 확인
   - 요구사항과 무관한 파일이 변경되지 않았는가?

2. 잔여물 확인
   - println, console.log, TODO, FIXME 같은 디버그 코드가 남아있지 않은가?
   - 주석 처리된 코드가 남아있지 않은가?

3. 기존 동작 영향
   - 기존 함수의 시그니처가 변경되지 않았는가?
   - 기존 테스트가 영향받지 않는가?
   - 삭제된 코드가 다른 곳에서 참조되고 있지 않은가?

4. SPEC 교차 확인 (SPEC.md 존재 시)
   - 이 태스크의 변경이 SPEC.md의 수용 기준(AC)과 일관되는가?
   - SPEC의 제약(Constraints)을 위반하지 않는가?
```

### 변경 영향 판정

```
✅ 깨끗함 — 모든 변경이 요구사항 범위 내
⚠️ 주의 — 관련 없는 변경이 있으나 무해함 (포맷팅, import 정리 등)
❌ 문제 — 의도하지 않은 동작 변경 또는 잔여 디버그 코드
```

## Phase 4: 정성적 코드 리뷰 (Qualitative Code Review)

git diff의 변경된 코드를 5가지 관점에서 검토하고, 문제 발견 시 자동 수정합니다.

### 체크 항목

1. 중복 (Duplication)
   - 변경 내 같은 로직이 2회 이상 반복되는가?
   - 기존 코드베이스에 동일/유사한 유틸이 이미 존재하는가?
   - Glob/Grep으로 기존 유틸 탐색 후 판단

2. 복잡도 (Complexity)
   - 더 단순한 구현이 가능한가?
   - 불필요한 추상화, 과도한 중첩, 우회적 로직이 있는가?
   - early return, 가드 절 등으로 단순화 가능한가?

3. 효율 (Efficiency)
   - 비효율적인 패턴이 있는가?
   - N+1 쿼리, 루프 안의 불필요한 연산, 과도한 객체 생성 등

4. 네이밍 (Naming)
   - 변수/함수/클래스 이름이 의도를 드러내는가?
   - 오해 소지 있는 이름, 축약어 남용이 있는가?
   - 프로젝트 기존 네이밍 컨벤션과 일치하는가?

5. 엣지케이스 (Edge Cases)
   - 빈 리스트, null, undefined 처리가 누락되어 있는가?
   - 경계값(0, 음수, 최대값)에서 정상 동작하는가?
   - 빈 문자열, 공백 문자열 구분이 필요한가?

### 자동 수정 규칙

문제 발견 시:
1. 즉시 코드 수정
2. Phase 4 재검증 (수정한 부분만)
3. 최대 3회 반복
4. 3회 초과 시 미수정 항목을 사용자에게 보고

### 보고 포맷

각 항목은 아래 형식으로 기록:
  - 관점: [중복/복잡도/효율/네이밍/엣지케이스]
  - 위치: 파일명:라인번호
  - 문제: 무엇이 문제인지
  - 조치: ✅ 수정 완료 / ⚠️ 수정 실패 (사유)

## 검증 결과 기록 (선택)

태스크 파이프라인에서 사용 시, 검증 결과를 태스크 파일에 기록할 수 있습니다:

```markdown
## 검증 이력
- 2026-02-18 1차: FAIL (2/5 미충족 — 정렬, 에러 처리)
- 2026-02-18 2차: PASS (5/5 충족)
```

이 기록은 선택적이며, 대화 내에서 결과를 보고하는 것만으로 충분합니다.
