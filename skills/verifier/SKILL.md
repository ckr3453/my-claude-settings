# 검증자 (Verifier) — 3단계 파이프라인 디스패처

검증은 반드시 **서브에이전트**로 실행한다.
메인 에이전트가 직접 검증하면 자기 코드에 대한 확증 편향이 생기고, 컨텍스트 오염으로 객관성이 떨어진다.

---

## 파이프라인 개요

```
Stage 1: 기계적 게이트 ($0, 메인 에이전트 직접 실행)
  → 빌드 + 테스트
  → 실패 시 즉시 중단 — LLM 서브에이전트 비용 절약

Stage 2: Verifier (code-reviewer 서브에이전트)
  → 요구사항 분해 → 증거 대조 → 변경 영향 → 잔여물 점검
  → FAIL 시 즉시 중단

Stage 3: Adversary (qa-expert 서브에이전트)
  → Verifier의 PASS 판정을 독립적으로 반박 시도
  → PASS를 FAIL로 내릴 수 있지만, FAIL을 PASS로 올릴 수 없음
```

각 단계를 통과해야 다음 단계로 진행한다. 어느 단계에서든 FAIL이면 전체 FAIL.

---

## Stage 1: 기계적 게이트

메인 에이전트가 직접 실행한다 (서브에이전트 불필요).

1. `references/detection-rules.md`의 탐지 표에 따라 프로젝트 빌드 도구를 감지한다
2. 변경 관련 테스트를 실행한다 (`git diff --name-only`로 대상 파일 수집)
3. 결과를 기록한다:
   - **빌드 성공 + 테스트 통과** → Stage 2로 진행
   - **빌드 실패 또는 테스트 실패** → 즉시 중단, FAIL 보고. 메인 에이전트가 수정 후 Stage 1부터 재실행

기계적 게이트 결과는 Stage 2 서브에이전트에게 전달한다 (중복 실행 방지).

---

## Stage 2: Verifier (서브에이전트)

### 컨텍스트 수집

서브에이전트에게 넘길 컨텍스트를 준비한다:

1. **검증 대상** — 아래 중 해당하는 것을 수집:
   - 태스크 파일 경로 (`.claude/tasks/` 하위)
   - 태스크 파일이 없으면: 사용자의 원래 요청을 요약
2. **프로젝트 루트 경로**
3. **PLAN 경로** — `active/{로드맵}/PLAN.md`가 존재하면 경로 포함
4. **Stage 1 결과** — 빌드/테스트 실행 결과 요약

### 서브에이전트 생성

```
Agent(
  subagent_type: "code-reviewer",
  description: "검증: {검증 대상 요약}",
  prompt: 아래 템플릿
)
```

**프롬프트 템플릿:**

```
당신은 독립적인 검증자(Verifier)입니다.
메인 에이전트가 작성한 코드를 객관적으로 검증하는 것이 임무입니다.
메인 에이전트의 판단에 동조하지 마세요. 증거만으로 판정하세요.

- 태스크 파일: {태스크 파일 경로 또는 "없음"}
- 사용자 요청: {태스크 파일이 없을 경우 원래 요청 요약}
- PLAN 경로: {PLAN 경로 또는 "없음"}

## Stage 1 (기계적 게이트) 결과
{빌드/테스트 실행 결과 — 이미 통과했으므로 재실행 불필요}

## 출력

```

### 결과 처리

- **FAIL** → 즉시 중단. 메인 에이전트가 FAIL 항목을 수정한 후 Stage 1부터 재실행.
- **PASS** → Verifier 보고서를 보존하고 Stage 3으로 진행.

---

## Stage 3: Adversary (서브에이전트)

Verifier가 PASS를 준 경우에만 실행한다.

### 서브에이전트 생성

```
Agent(
  subagent_type: "qa-expert",
  description: "반박 검증: {검증 대상 요약}",
  prompt: 아래 템플릿
)
```

**프롬프트 템플릿:**

```
당신은 Adversary(반박 검증자)입니다.
앞서 Verifier가 이 코드를 PASS 판정했습니다. 당신의 임무는 그 판정에 구멍이 없는지 독립적으로 검증하는 것입니다.
Verifier의 판단을 신뢰하지 마세요. 직접 코드를 읽고 판단하세요.

## 검증 대상
- 프로젝트 루트: {프로젝트 루트 경로}
- 태스크 파일: {태스크 파일 경로 또는 "없음"}
- 사용자 요청: {태스크 파일이 없을 경우 원래 요청 요약}

## Verifier 보고서
{Stage 2 Verifier의 전체 보고서를 여기에 붙여넣기}

## 반박 프로토콜
`/Users/ckr/.claude/skills/verifier/adversary-prompt.md` 파일을 읽고 그 프로토콜을 따라 반박 검증을 수행하세요.

## 출력
반박 검증 결과 보고서를 반환하세요. 커밋은 하지 마세요.
```

### 결과 처리

- **Adversary가 FAIL 항목을 발견** → 전체 FAIL. 메인 에이전트가 수정 후 Stage 1부터 재실행.
- **Adversary가 반박 실패 (PASS 유지)** → 최종 PASS.

---

## 최종 처리

### PASS 시: 자동 커밋

1. `git diff --stat`으로 변경 파일 확인
2. `git log --oneline -5`로 커밋 스타일 감지
3. 커밋 메시지 생성 → `git add` + `git commit` 실행

### 결과 보고

사용자에게 아래를 전달한다:

1. **Stage 1 결과** (빌드/테스트)
2. **Stage 2 Verifier 보고서** (그대로 전달, 재해석 금지)
3. **Stage 3 Adversary 보고서** (그대로 전달, 재해석 금지)
4. **최종 판정**: 3단계 모두 통과 시 PASS, 하나라도 실패 시 FAIL

---

## 금지 사항

- 메인 에이전트가 검증 프로토콜을 직접 실행하지 않는다 (Stage 1 제외)
- 서브에이전트의 FAIL 판정을 메인 에이전트가 PASS로 뒤집지 않는다
- Adversary의 FAIL 판정을 Verifier의 PASS로 무효화하지 않는다
- 검증 결과를 요약하거나 긍정적으로 재해석하지 않는다
