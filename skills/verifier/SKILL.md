# 검증자 (Verifier)

작업 결과가 원래 요구사항을 100% 충족하는지 검증합니다.

**검증은 모델이 판정한다. 실행은 워크플로우가 강제한다.**
**요구사항 충족 여부가 핵심이며, 기술 검증(테스트)은 충족의 필수 증거다.**

---

## AI 행동 규칙 (반드시 준수)

### 1. 검증 기준 확보 (폴백 계층)

검증할 기준을 아래 순서로 탐색합니다:

```
0순위: 태스크 파일의 검증 체크리스트 (사용자 승인 완료)
       → 태스크 시작 시 생성되어 사용자가 승인한 체크리스트
       → 있으면 Phase 1(분해)을 건너뛰고 바로 Phase 2(증거 대조)로 진입

1순위: 태스크 파일의 완료 기준
       → 검증 체크리스트가 없으면 완료 기준을 사용
       → 단, 검증 불가능한 항목("구현 완료" 등)은 2순위로 폴백

2순위: 태스크 파일의 목표에서 도출
       → 완료 기준이 없거나 부실하면 목표 섹션에서 구체적 기준 도출

3순위: 사용자의 원래 요청에서 도출
       → 태스크 파일 자체가 없는 일상 작업
       → 대화 내 사용자 요청을 다시 읽고 원자적 검증 항목으로 분해
```

**어떤 상황에서든 검증 기준을 확보한 후에만 검증을 시작합니다.**

### 2. 4단계 검증 프로토콜

검증은 반드시 아래 4단계를 순서대로 실행합니다:

```
Phase 1: 요구사항 분해 (Requirement Decomposition)
  → 확보한 기준을 원자적(atomic) 검증 항목으로 분해
  → 각 항목은 "존재한다/동작한다/반환한다" 같은 검증 가능한 동사로 끝남

Phase 2: 증거 기반 대조 (Evidence-Based Verification)
  → 각 항목에 대해 코드에서 증거를 찾아 읽음
  → 판정 부여 (기준은 섹션 5 참조)
  → 증거 기록: 파일명:라인번호

Phase 3: 변경 영향 리뷰 (Change Impact Review)
  → git diff로 모든 변경사항 확인
  → 요구사항과 무관한 변경, 디버그 코드, 기존 동작 영향 점검

Phase 4: 잔여물 점검 (Residue Check)
  → git diff의 변경된 코드만 대상
  → 자동 수정 대상: 디버그 코드 잔존(println, console.log), 미사용 import/변수
  → 이번 변경에서 새로 추가된 TODO만 대상 (기존 TODO는 건드리지 않음)
  → 보고만 (수정 안 함): 중복, 복잡도, 효율, 네이밍, 엣지케이스
```

### 2.5 PLAN 교차 확인

`active/{로드맵}/PLAN.md`가 존재하면 Phase 3(변경 영향 리뷰)에서 상위 맥락 교차 확인을 추가로 수행합니다:

- 이 태스크의 변경이 PLAN.md의 요구사항과 일관되는가?
- PLAN의 제약(범위 밖)을 위반하지 않는가?
- 교차 확인 결과를 변경 영향 섹션에 1~2줄로 기록


### 3. 테스트 존재 여부는 필수 증거

**검증 항목에 대한 테스트가 존재하고 통과해야 PASS다.**

verifier는 테스트를 직접 작성하지 않는다. 대신:

1. 각 검증 항목에 대해 해당 동작을 검증하는 테스트가 존재하는지 탐색한다
2. 테스트가 존재하면 실행하여 통과 여부를 확인한다
3. 컴파일 검증은 항상 실행한다

**판정 규칙:**
- 테스트 존재 + 통과 → 해당 항목 PASS 가능
- 테스트 존재 + 실패 → 해당 항목 FAIL
- 테스트 미존재 → 해당 항목 ⚠️ PARTIAL (코드 증거만으로는 PASS 불가, "테스트 미작성" 명시)

**예외: 테스트 불필요 항목**
- 단순 설정 변경, 의존성 추가, 리소스 파일 등 테스트로 검증할 수 없는 항목은 코드 증거만으로 PASS 가능
- 이 경우 "테스트 불필요 (사유)" 명시

탐지 규칙과 실행 규칙은 `references/detection-rules.md` 참조.

### 4. 검증 결과 보고

검증 완료 후 아래 형식으로 결과를 보고합니다:

```markdown
## 검증 결과

### 검증 기준 출처
{0순위~3순위 중 어디서 기준을 확보했는지 명시}

### 요구사항 대조
| # | 검증 항목 | 판정 | 증거 |
|---|----------|------|------|
| 1 | ... | ✅ PASS | file.kt:45 |
| 2 | ... | ❌ FAIL | 미구현 |
| 3 | ... | ⚠️ PARTIAL | 부분 구현 (상세) |

### 변경 영향
- 변경 파일 N개, 요구사항 관련 여부
- 디버그 코드 잔존 여부
- 기존 동작 영향 여부

### 기술 검증 (탐지: {프레임워크})
- {명령어}: {결과}

### 잔여물 점검
| # | 유형 | 파일:라인 | 문제 | 조치 |
|---|------|----------|------|------|
| 1 | 디버그 코드 | service.kt:30 | println 잔존 | ✅ 제거 |
| 2 | 미사용 import | util.kt:5 | 미사용 import 잔존 | ✅ 제거 |

(문제 없으면 "잔여물 없음" 한 줄로 갈음)

### 코드 품질 소견 (보고만, 수정 안 함)
- 중복/복잡도/효율/네이밍/엣지케이스 중 발견된 사항만 기록
- 없으면 생략

### 최종 판정: ✅ PASS / ❌ FAIL (N개 항목 미충족)
```

### 5. 판정 기준

FAIL 또는 PARTIAL이 1개라도 있으면 최종 판정은 FAIL.

| 코드 증거 | 테스트 존재 | 테스트 결과 | 최종 판정 |
|-----------|-----------|-----------|----------|
| PASS | 존재 | PASS | ✅ PASS |
| PASS | 존재 | FAIL | ❌ FAIL (테스트 실패) |
| PASS | 미존재 | - | ⚠️ PARTIAL (테스트 미작성) |
| FAIL | 존재 | PASS | ⚠️ PARTIAL (코드 부재, 테스트만 존재) |
| FAIL | 미존재 | - | ❌ FAIL |

### 6. 판정하지 않는 것

verifier는 아래에 대해 판정하지 않습니다:

- 코드 스타일/포맷팅 (린터의 영역)
- 아키텍처 적절성 (설계 단계의 영역)
- 테스트 커버리지 수치 (요구사항에 명시되지 않은 한)

**Phase 1-3은 "요구사항이 충족되었는가"를, Phase 4는 "잔여물(디버그 코드, 미사용 코드)이 없는가"를 점검합니다.**

### 7. PASS 후 자동 커밋

검증 PASS 후 자동으로 커밋합니다:

1. `git diff --stat`으로 변경 파일 확인
2. `git log --oneline -5`로 커밋 스타일 감지
3. 커밋 메시지 생성 → `git add` + `git commit` 실행

### 8. 에러 복구 및 재시도

#### 증거를 찾을 수 없는 항목이 다수일 때

1. 검증을 중단하지 않고, 찾을 수 있는 항목부터 모두 검증
2. 증거 부재 항목은 FAIL로 판정하되, "미구현 또는 탐색 범위 밖" 명시
3. 최종 보고에서 "증거 탐색 범위"를 기록 (어디를 찾았는지)

#### 기술 검증 도구 실행 실패

1. 1회 재시도 (환경 문제일 수 있음)
2. 재시도 실패 → 기술 검증 섹션에 "실행 실패: {에러}" 기록
3. Phase 1-3 결과만으로 최종 판정 (실행 실패한 기술 검증은 판정에서 제외)

#### FAIL 시 가이드라인

- 같은 항목이 반복 FAIL → 접근 방식을 바꿔서 재시도
- 진전이 없으면 해당 항목만 사용자에게 보고하고 나머지는 계속
