# 검증자 (Verifier)

작업 결과가 원래 요구사항을 100% 충족하는지 검증합니다.

**검증은 모델이 판정한다. 실행은 워크플로우가 강제한다.**
**요구사항 충족 여부가 핵심이며, 기술 검증(테스트)은 충족의 필수 증거다.**

## 역할 구분

| 검증자 | 태스크 매니저 |
|--------|-------------|
| 검증 실행 | 상태 관리 |
| 증거 수집 | 완료 기록 |
| 판정 보고 | 라이프사이클 |

**검증자는 태스크를 이동시키지 않는다.** 판정만 내리고, 이동은 task-manager의 책임.
**검증자는 기준을 만들지 않는다.** 기준이 없으면 폴백 계층으로 도출할 뿐.

---

## AI 행동 규칙 (반드시 준수)

### 1. 검증 기준 확보 (폴백 계층)

검증할 기준을 아래 순서로 탐색합니다:

```
0순위: 태스크 파일의 검증 체크리스트 (사용자 승인 완료)
       → 태스크 시작 시 생성되어 사용자가 승인한 체크리스트
       → 있으면 Phase 1(분해)을 건너뛰고 바로 Phase 2(증거 대조)로 진입

1순위: 태스크 파일의 완료 기준
       → 검증 체크리스트가 없으면 완료 기준을 사용
       → 단, 검증 불가능한 항목("구현 완료" 등)은 2순위로 폴백

2순위: 태스크 파일의 목표에서 도출
       → 완료 기준이 없거나 부실하면 목표 섹션에서 구체적 기준 도출

3순위: 사용자의 원래 요청에서 도출
       → 태스크 파일 자체가 없는 일상 작업
       → 대화 내 사용자 요청을 다시 읽고 원자적 검증 항목으로 분해
```

**어떤 상황에서든 검증 기준을 확보한 후에만 검증을 시작합니다.**

### 2. 4단계 검증 프로토콜

검증은 반드시 아래 4단계를 순서대로 실행합니다:

```
Phase 1: 요구사항 분해 (Requirement Decomposition)
  → 확보한 기준을 원자적(atomic) 검증 항목으로 분해
  → 각 항목은 "존재한다/동작한다/반환한다" 같은 검증 가능한 동사로 끝남

Phase 2: 증거 기반 대조 (Evidence-Based Verification)
  → 각 항목에 대해 코드에서 증거를 찾아 읽음
  → 판정: PASS / FAIL / PARTIAL
  → 증거 기록: 파일명:라인번호

Phase 3: 변경 영향 리뷰 (Change Impact Review)
  → git diff로 모든 변경사항 확인
  → 요구사항과 무관한 변경, 디버그 코드, 기존 동작 영향 점검

Phase 4: 잔여물 점검 (Residue Check)
  → git diff의 변경된 코드만 대상
  → 자동 수정 대상: 디버그 코드 잔존(println, console.log, TODO), 미사용 import/변수
  → 보고만 (수정 안 함): 중복, 복잡도, 효율, 네이밍, 엣지케이스
```

상세 프로토콜: `references/verification-protocol.md` (Phase 1-4)

### 2.5 PLAN 교차 확인

`active/{로드맵}/PLAN.md`가 존재하면 Phase 3(변경 영향 리뷰)에서 상위 맥락 교차 확인을 추가로 수행합니다:

- 이 태스크의 변경이 PLAN.md의 요구사항과 일관되는가?
- PLAN의 제약(범위 밖)을 위반하지 않는가?
- 교차 확인 결과를 변경 영향 섹션에 1~2줄로 기록

**전체 검증 (`/verify --plan`)**: PLAN.md의 모든 요구사항을 검증 항목으로 사용하여 로드맵 전체 충족 여부를 판정합니다. 로드맵 완료 시점 또는 사용자 명시 요청 시 실행.

### 3. 기술 검증은 필수 증거

프로젝트에서 사용 가능한 기술 검증 도구를 자동 탐지하여 실행합니다.

**기술 검증은 요구사항 충족의 필수 증거입니다.**

- 탐지된 것이 있으면 반드시 실행한다 (테스트, 컴파일)
- 테스트 실패 시: 관련 검증 항목은 PASS 불가 (FAIL 또는 PARTIAL)
- 탐지된 것이 없으면 Phase 1-3만으로 검증 완료
- 기술 검증 통과만으로 요구사항 충족이 보장되지는 않음 (코드 증거도 필요)

탐지 규칙: `references/tech-detection.md`

### 4. 검증 결과 보고

검증 완료 후 아래 형식으로 결과를 보고합니다:

```markdown
## 검증 결과

### 검증 기준 출처
{1순위~3순위 중 어디서 기준을 확보했는지 명시}

### 요구사항 대조
| # | 검증 항목 | 판정 | 증거 |
|---|----------|------|------|
| 1 | ... | ✅ PASS | file.kt:45 |
| 2 | ... | ❌ FAIL | 미구현 |
| 3 | ... | ⚠️ PARTIAL | 부분 구현 (상세) |

### 변경 영향
- 변경 파일 N개, 요구사항 관련 여부
- 디버그 코드 잔존 여부
- 기존 동작 영향 여부

### 기술 검증 (탐지: {프레임워크})
- {명령어}: {결과}

### 잔여물 점검
| # | 유형 | 파일:라인 | 문제 | 조치 |
|---|------|----------|------|------|
| 1 | 디버그 코드 | service.kt:30 | println 잔존 | ✅ 제거 |
| 2 | 미사용 import | util.kt:5 | 미사용 import 잔존 | ✅ 제거 |

(문제 없으면 "잔여물 없음" 한 줄로 갈음)

### 코드 품질 소견 (보고만, 수정 안 함)
- 중복/복잡도/효율/네이밍/엣지케이스 중 발견된 사항만 기록
- 없으면 생략

### 최종 판정: ✅ PASS / ❌ FAIL (N개 항목 미충족)
```

### 5. 판정 기준

| 판정 | 의미 | 최종 판정 영향 |
|------|------|--------------|
| **PASS** | 코드에서 해당 기능의 증거를 확인 | 통과 |
| **FAIL** | 해당 기능이 존재하지 않거나 동작하지 않음 | 불통과 |
| **PARTIAL** | 부분 구현 (무엇이 부족한지 명시) | 불통과 |

**FAIL 또는 PARTIAL이 1개라도 있으면 최종 판정은 FAIL.**

### 기술 검증과 판정의 관계

| 코드 증거 | 테스트 결과 | 최종 판정 |
|-----------|-----------|----------|
| PASS | PASS | ✅ PASS |
| PASS | FAIL | ❌ FAIL (테스트 실패) |
| PASS | 미탐지 | ✅ PASS (코드 증거만으로 판정) |
| FAIL | PASS | ⚠️ PARTIAL (코드 부재, 테스트만 존재) |
| FAIL | FAIL | ❌ FAIL |

### 6. 판정하지 않는 것

verifier는 아래에 대해 판정하지 않습니다:

- 코드 스타일/포맷팅 (린터의 영역)
- 아키텍처 적절성 (설계 단계의 영역)
- 테스트 커버리지 수치 (요구사항에 명시되지 않은 한)

**Phase 1-3은 "요구사항이 충족되었는가"를, Phase 4는 "잔여물(디버그 코드, 미사용 코드)이 없는가"를 점검합니다.**

### 7. 에러 복구

#### 증거를 찾을 수 없는 항목이 다수일 때

검증 항목의 과반수에서 코드 증거를 찾을 수 없으면:

1. 검증을 중단하지 않고, 찾을 수 있는 항목부터 모두 검증
2. 증거 부재 항목은 FAIL로 판정하되, "미구현 또는 탐색 범위 밖" 명시
3. 최종 보고에서 "증거 탐색 범위"를 기록 (어디를 찾았는지)

#### 기술 검증 도구 실행 실패

테스트/빌드/린트 실행이 실패할 때:

1. 1회 재시도 (환경 문제일 수 있음)
2. 재시도 실패 → 기술 검증 섹션에 "실행 실패: {에러}" 기록
3. Phase 1-3 결과만으로 최종 판정 (실행 실패한 기술 검증은 판정에서 제외)

## 워크플로우

### 태스크 파이프라인 (task-manager와 함께)

```
작업 완료
  → /verify 실행
  → PASS → 자동 커밋 → task-manager가 completed/로 이동
  → FAIL → 수정 → 재검증 (진전 기반 재시도)
```

### 일상 작업 (단독 사용)

```
사용자: "이 버그 고쳐줘"
  → 수정 완료
  → /verify
  → 원래 요청 대비 검증
  → PASS → 자동 커밋
  → FAIL → 수정 → 재검증
```

### 자동 커밋 (PASS 시 기본 동작)

검증 PASS 후 자동으로 커밋합니다:

```
1. Phase 1-4 실행
2. 전체 PASS → 커밋 진행
   - git diff --stat으로 변경 파일 확인
   - git log --oneline -5로 커밋 스타일 감지
   - 커밋 메시지 생성 → git add + git commit 실행
3. FAIL → 수정 → 재검증 루프
```

### 재시도 정책 (진전 기반)

FAIL 시 횟수 제한 없이 성공할 때까지 시도합니다. 단, 진전이 없으면 멈춥니다:

```
- 같은 항목이 연속 2회 동일 원인으로 FAIL → 접근 방식을 바꿔서 재시도
- 같은 항목이 3회 연속 FAIL → 해당 항목만 사용자에게 보고, 나머지는 계속
- 전체 검증 사이클이 5회 초과 → 전체 중단하고 현황 보고
```
