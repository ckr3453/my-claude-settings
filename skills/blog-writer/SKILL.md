# 블로그 라이터 (Blog Writer)

코딩 작업 경험을 캐주얼 에세이 스타일의 기술 블로그 포스트로 변환한다.

**핵심 원칙**: 교과서가 아니라 개발자의 체험담. "~했더니 ~되더라" 스타일.

## 사용하지 않는 경우

이 스킬은 아래 용도로 사용하지 않는다:

- 기술 문서/API 레퍼런스 작성
- 뉴스/리포트 형식의 글
- 코드 변경 없는 순수 이론 정리
- 튜토리얼/가이드 형식 ("~하세요" 체)

---

## 포스트 유형 (4가지)

| 유형 | 핵심 | 신호 |
|------|------|------|
| **삽질기** | 문제 해결 과정을 시간순으로 | 에러, 디버깅, fix 커밋, revert |
| **의사결정기** | 왜 이 기술을 선택했는지 | A vs B, 설정/의존성 변경 |
| **TIL** | 짧고 밀도 있는 하나의 발견 | 새로 배운 것, 단일 파일 변경 |
| **회고** | 기간/프로젝트를 돌아보며 성장 정리 | 프로젝트 마무리, 대규모 변경 |

상세 템플릿: `references/post-type-templates.md`

## AI 행동 규칙 (반드시 준수)

### 1. 모드 선택

트리거에 따라 모드가 자동 결정된다:

| 트리거 | 모드 |
|--------|------|
| `/blog`, "블로그 써줘", "오늘 작업 블로그로" | **Mode A** (git 분석) |
| `/blog-interview`, "블로그 글감" | **Mode B** (인터뷰) |

- Mode A는 현재 디렉토리가 git 저장소일 때만 실행
- git 저장소가 아니면 자동으로 Mode B로 전환하고 사용자에게 안내

### 2. 공통 워크플로우

모든 모드에서 아래 흐름을 따른다:

```
글감 수집 → 주제 후보 2~3개 제시 → [사용자 선택] ①
→ 포스트 유형 확인 → [사용자 승인] ②
→ 초안 생성 → [사용자 리뷰] ③
→ 수정 반영 → Jekyll 파일 생성
```

**사용자 확인 필수 지점 3곳:**

| # | 시점 | 방법 |
|---|------|------|
| ① | 주제 선택 | `AskUserQuestion` — 후보 중 택 1 |
| ② | 포스트 유형 확인 | `AskUserQuestion` — 추천 유형 + 대안 |
| ③ | 초안 리뷰 | 초안을 대화에 출력 후 피드백 요청 |

**사용자 확인 없이 다음 단계로 진행하지 않는다.**

### 3. Mode A: Git 분석

현재 프로젝트의 git 이력을 분석하여 글감을 추출한다.

**실행 순서:**

1. `git log --oneline --since="3 days ago" --no-merges`로 최근 커밋 수집
2. `git diff --stat`으로 주요 변경 파일 파악
3. 핵심 파일의 diff를 읽고 변경 패턴 분석
4. 커밋 메시지 키워드로 유형 추정
5. **주제 후보 2~3개** 생성하여 사용자에게 제시

각 후보는 다음 정보를 포함:

```
[유형] 제목 후보
- 근거: 어떤 커밋/변경에서 도출했는지
- 핵심 변경: 글의 중심이 될 내용
```

상세 프로토콜: `references/git-analysis-protocol.md`

### 4. Mode B: 인터뷰

대화를 통해 글감을 발굴한다.

**실행 순서:**

1. `AskUserQuestion`으로 오프닝 질문 (최근 기억에 남는 개발 경험)
2. 응답에서 포스트 유형 추정
3. 유형별 후속 질문 3~5개를 `AskUserQuestion`으로 **1개씩** 순차 진행
4. 답변을 종합하여 **주제 후보 2~3개** 생성

**인터뷰 원칙:**
- 한 번에 1개 질문만
- 사용자 답변을 그대로 소재로 활용
- 최소 3개, 최대 5개 질문 후 마무리

상세 흐름: `references/interview-flow.md`

### 5. 초안 생성

사용자가 주제와 유형을 확인하면 초안을 생성한다.

**초안 생성 규칙:**

1. `references/writing-style-guide.md`의 문체 규칙 적용
2. `references/post-type-templates.md`의 해당 유형 골격 사용
3. `references/jekyll-format.md`의 front matter 포맷 적용
4. 초안 전체를 대화에 출력 (파일 생성 아님)
5. 사용자 피드백 대기

**초안에 반드시 포함:**
- Jekyll front matter (title, categories, date, last_modified_at, toc, toc_sticky, excerpt)
- 유형별 골격에 맞는 본문 구조
- 캐주얼 에세이 문체 ("~했다", "~더라")
- 코드 블록 (관련 코드가 있는 경우, 10줄 이내)

**초안에 포함하지 않는 것:**
- 이모지/이모티콘
- "~합니다" 경어체
- "~에 대해 알아보겠습니다" 같은 도입부
- 이미지 플레이스홀더

### 6. 수정 사이클

사용자 피드백에 따라 초안을 수정한다.

- **부분 수정 요청**: 해당 부분만 수정하여 전체 초안 재출력
- **톤/방향 수정**: 문체 가이드 재적용하여 재작성
- **수정 3회 이상**: 방향을 재확인
  ```
  "수정이 3회를 넘었습니다. 전체 방향을 다시 잡을까요,
   아니면 현재 버전에서 세부 조정만 할까요?"
  ```
- **"됐어" / "이대로"**: 수정 종료, 파일 생성 단계로

### 7. Jekyll 파일 생성

초안이 확정되면 Jekyll 포맷의 마크다운 파일을 생성한다.

**블로그 저장소:** 파일 생성 시점에 `AskUserQuestion`으로 경로를 1회 확인한다.

#### 기존 카테고리에 글 쓰기 (포스트 1개만 생성)

1. 카테고리에 맞는 하위 디렉토리 선택 (`_posts/{카테고리 경로}/`)
2. 파일명: `YYYY-MM-DD-slug.md` (slug는 언더스코어 구분)
3. 파일 내용: front matter + 확정된 본문

#### 새 카테고리 만들기 (사용자 확인 필수, 3개 파일 생성/수정)

1. **아카이브 페이지 생성**: `_pages/categories/{섹션}/category-{name}.md`
2. **사이드바 수정**: `_includes/nav_list_main`에 카테고리 항목 추가
3. **포스트 작성**: `_posts/{섹션}/{name}/YYYY-MM-DD-slug.md`

새 카테고리가 필요한 글감이면, 주제 선택(①) 시점에 사용자에게 카테고리 생성 여부를 함께 확인한다.

상세 포맷 및 템플릿: `references/jekyll-format.md`

### 8. 에러 복구

#### git 저장소가 아닌 경우 (Mode A)

```
"현재 디렉토리가 git 저장소가 아닙니다.
 인터뷰 모드로 전환할까요?"
```
→ 사용자 동의 시 Mode B로 전환

#### git 커밋이 없거나 부족한 경우

```
"최근 3일간 커밋이 부족합니다.
 (1) 기간을 7일/2주로 넓힐까요?
 (2) 인터뷰 모드로 전환할까요?"
```

#### 블로그 저장소 경로가 유효하지 않은 경우

```
"지정한 경로에 _posts 디렉토리가 없습니다.
 경로를 다시 확인해주세요."
```
→ `_posts` 디렉토리가 없으면 생성 여부를 사용자에게 확인

#### 초안 생성 중 문맥 부족

소재가 충분하지 않아 초안 작성이 어려우면:
1. 부족한 정보를 구체적으로 안내
2. 추가 질문 2~3개로 보충
3. 보충 후 재시도

## 워크플로우 요약

### Mode A (실작업 기반)

```
/blog
  → git log/diff 분석
  → 주제 후보 2~3개 제시
  → [사용자 선택] ①
  → 포스트 유형 확인
  → [사용자 승인] ②
  → 초안 생성 (대화에 출력)
  → [사용자 리뷰] ③
  → 수정 반영 (필요 시 반복)
  → Jekyll 파일 생성
```

### Mode B (인터뷰 기반)

```
/blog-interview
  → 오프닝 질문
  → 유형 감지
  → 후속 질문 3~5개
  → 주제 후보 2~3개 제시
  → [사용자 선택] ①
  → 이후 Mode A와 동일
```
