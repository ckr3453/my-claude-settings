# Mode A: Git 분석 프로토콜

## 개요

현재 프로젝트의 git 이력을 분석하여 블로그 글감을 자동 추출한다.

## 분석 단계

### Step 1: 최근 커밋 수집

```bash
git log --oneline --since="3 days ago" --no-merges
```

- 기본 범위: 최근 3일
- 커밋이 5개 미만이면 범위를 7일로 확장
- 커밋이 0개이면 사용자에게 기간 확인

### Step 2: 주요 변경 파일 파악

```bash
git diff --stat HEAD~{N}..HEAD
```

- 변경량이 많은 파일 Top 5 추출
- 테스트 파일 변경 동반 여부 확인
- 설정 파일 변경 여부 확인

### Step 3: 핵심 diff 분석

```bash
git diff HEAD~{N}..HEAD -- {주요 파일}
```

상위 3개 파일의 diff를 읽고 다음을 파악:

| 분석 항목 | 의미 |
|----------|------|
| 코드 추가 위주 | 새 기능 구현 → 의사결정기/TIL 후보 |
| 수정/삭제 위주 | 버그 수정/리팩토링 → 삽질기 후보 |
| 설정/의존성 변경 | 기술 스택 변경 → 의사결정기 후보 |
| 테스트 추가/수정 | 테스트 관련 경험 → 삽질기/TIL 후보 |
| 다수 파일 대규모 변경 | 프로젝트 단위 작업 → 회고 후보 |

### Step 4: 커밋 메시지 패턴 분석

| 커밋 메시지 키워드 | 추정 유형 |
|-------------------|----------|
| fix, bug, hotfix, resolve | 삽질기 |
| feat, add, implement | 의사결정기 또는 TIL |
| refactor, clean, improve | 의사결정기 |
| chore, config, setup | 의사결정기 |
| test, spec | TIL 또는 삽질기 |
| revert | 삽질기 (강한 신호) |

### Step 5: 주제 후보 생성

분석 결과를 종합하여 **2~3개 주제 후보**를 생성한다.

각 후보에 포함할 정보:

```
1. [유형: 삽질기] Spring Security 설정 삽질기
   - 근거: fix 커밋 3개, SecurityConfig.java 대규모 수정
   - 핵심 변경: CORS 설정 + JWT 필터 순서 변경

2. [유형: TIL] Kotlin sealed class로 에러 처리하기
   - 근거: ErrorResponse.kt 신규 생성, 관련 핸들러 수정
   - 핵심 변경: sealed class 도입으로 에러 타입 통합
```

## 분석 제외 대상

다음은 글감 후보에서 제외:

- `.gitignore`, `README.md` 단독 변경
- 자동 생성 파일 (lock 파일, 빌드 산출물)
- 포맷팅/린트 자동 수정만 있는 커밋
- merge commit

## 글감이 부족할 때

git 이력에서 명확한 글감이 나오지 않을 때:

1. 사용자에게 알림: "최근 커밋에서 뚜렷한 글감을 찾기 어렵습니다"
2. 대안 제시:
   - 기간 확장 (2주, 1개월)
   - Mode B(인터뷰)로 전환 제안
   - 특정 커밋/브랜치 지정 요청

## 복수 프로젝트

현재 작업 디렉토리의 git 저장소만 분석한다.
다른 프로젝트를 분석하려면 사용자가 경로를 지정해야 한다.
