# 캐주얼 에세이 문체 가이드

## 핵심 원칙

**"정돈된 캐주얼 — 블로그 에세이"**

친구한테 말하는 것도 아니고, 논문을 쓰는 것도 아니다.
글로 쓰여진 느낌은 유지하되, 자연스럽고 가벼운 톤.

## 톤 스펙트럼

```
논문/경어체 ← ──── ★ 여기 ──── → 카톡/구어체
              정돈된 캐주얼
```

| 너무 딱딱함 (X) | 적절함 (O) | 너무 가벼움 (X) |
|----------------|-----------|----------------|
| 확인하였습니다 | 확인해봤다 | 확인해봤는데 와 |
| 생산성이 향상되었다 | 생산성이 눈에 띄게 올라갔다 | 생산성이 미치도록 올라갔다 |
| 부족함을 인지하게 되었다 | 부족하다는걸 느꼈다 | 부족하구나 진짜.. |
| 해당 문제를 해결하였다 | 결국 해결했다 | 됐다! 드디어! |

## 문체 규칙

### 어미

| 사용 | 지양 |
|------|------|
| ~했다 / ~더라 / ~됐다 | ~합니다 / ~입니다 (경어체) |
| ~인것 같다 / ~해봤는데 | ~하시면 됩니다 (지시체) |
| ~였는데 / ~거든 | ~해야 합니다 (당위체) |
| ~하다보니 / ~해서 | ~하였습니다 (논문체) |

**기본 어미**: `~했다`, `~더라`, `~됐다` (해라체 + 경험 서술)

`~거든`은 문단당 1회 이하로 사용. 남용하면 구어체처럼 들린다.

### 1인칭 시점

- "나는" 대신 생략하거나 "이번에" "그동안" 같은 시간 표현으로 시작
- 적절히 "나"를 섞되 매 문장마다 쓰지 않음
- 예: "프로젝트에서 Redis를 쓰고 있었는데, 갑자기 메모리가 터졌다."

### 감정 표현

감정은 **상황 서술로 간접 표현**한다. 감탄사를 직접 쓰지 않는다.

| 지양 (직접 감탄) | 권장 (간접 서술) |
|-----------------|-----------------|
| "뭐지?" | "당황스러웠다" |
| "드디어!" | "결국 해결했다" |
| "왜 안 되는 거야" | "원인을 알 수 없었다" |

### 절대 사용 금지

- 인터넷 감탄사: `ㅋㅋ`, `ㅎㅎ`, `ㅜㅜ`, `ㄷㄷ`, `;` 등
- 이모지/이모티콘
- 과도한 구어체: "야근 각", "뭘 했냐고 물으면"

### 허용하는 캐주얼 요소

- `..` 여운 — 적절히 사용 (문단당 1~2회 이하)
- 괄호 부연설명 — 자연스럽게 사용 가능
  - 예: "(사실 5권도 적긴 한데 현실적으로 잡아본 숫자)"
- 띄어쓰기 약간 느슨 — `있는데` 대신 `있었는데`, `것 같다` 대신 `것같다` 허용

### 문장 길이

- **짧은 문장 위주**: 한 문장에 하나의 정보
- 긴 설명이 필요하면 짧은 문장 여러 개로 나눔
- 평균 20~40자 / 문장

### 단락 구성

- 한 단락: 2~4문장
- 단락 사이 빈 줄 1개
- 핵심 포인트는 단독 문장으로 분리

## 구조적 문체 패턴

### 삽질 → 해결 패턴 (가장 많이 사용)

```
[상황 설정] 어떤 작업을 하고 있었다
[문제 발생] 이런 에러가 발생했다
[첫 번째 시도] 검색해서 이걸 시도해봤다
[실패] 해결되지 않았다
[두 번째 시도] 다른 방법을 시도했다
[해결] 결국 해결했다. 원인은 이거였다
[교훈] 앞으로는 이렇게 해야겠다
```

### 의사결정 패턴

```
[배경] 이런 상황이었다
[선택지] A와 B가 있었다
[고민 과정] A는 이런 점이 좋고, B는 저런 점이 좋았다
[결정] 결국 A를 선택했다
[이유] 이유는 이거다
[결과] 실제로 써보니 이랬다
```

### TIL 패턴

```
[계기] 이걸 하다가 알게 됐다
[내용] 핵심은 이거다
[예시] 코드로 보면 이렇다
[의미] 이걸 알면 이런 데서 쓸 수 있다
```

## 코드 블록 사용 규칙

- 코드 전에 한 줄 설명: "이렇게 바꿔봤다."
- 코드 후에 결과 설명: "이러니까 에러가 사라졌다."
- 긴 코드보다 핵심 부분만 발췌 (10줄 이내 권장)
- 변경 전/후를 보여줄 때는 두 블록으로 분리

```
이렇게 되어 있던 코드를:

​```java
// before
​```

이렇게 바꿨다:

​```java
// after
​```
```

## 제목 작성 규칙

- **메인 제목**: 키워드 포함 + 간결하게
  - 좋은 예: "그동안의 회고 그리고 2026년 각오"
  - 좋은 예: "Spring Boot 테스트가 10분 걸리길래 고쳐봤다"
  - 나쁜 예: "Spring Boot 테스트 최적화 방법" (가이드/튜토리얼 톤)
  - 나쁜 예: "Spring Boot 테스트에 대해 알아보자" (도입부 어투)
- **소제목(##)**: 이야기 흐름을 보여주는 서술형
  - 좋은 예: "## 서버팀 1인 체제가 됐다"
  - 나쁜 예: "## 서버팀 현황"

## 분량 가이드

| 포스트 유형 | 권장 분량 |
|------------|----------|
| 삽질기 | 800~1500자 |
| 의사결정 | 1000~2000자 |
| TIL | 500~800자 |
| 회고 | 1500~2500자 |

**분량보다 밀도가 중요.** 늘리려고 억지로 늘리지 않는다.
