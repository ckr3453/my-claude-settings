# 병렬 실행 및 동시성 규칙

디렉토리 기반 태스크 구조를 활용한 경량 병렬 처리 규칙.

---

## 동시성 규칙 (멀티세션)

### 안전한 동시 작업 (여러 세션 OK)

- 태스크 작업 (각 세션이 자기 파일만 mv)
- /brief 실행 (읽기 전용)

### 단일 세션에서만 수행

- Phase 전환 (아카이빙 + 새 태스크 생성)
- 로드맵 갱신
- pending/에 새 태스크 파일 일괄 생성

### 태스크 선점 충돌 방지

각 세션은 `pending/`에서 태스크를 선택한 즉시 `in_progress/`로 이동합니다.
다른 세션은 `in_progress/`에 있는 파일을 건너뜁니다.

---

## 판단 기준: 3단계 티어

### 1단계: 순차 처리 (기본, 토큰 1x)

**대부분의 경우 이것이 최적.** 아래 조건에 해당하면 순차 처리:

- 실행 가능한 독립 태스크가 2개 이하
- L 복잡도 태스크
- 같은 도메인/파일을 수정하는 태스크들
- 디버깅, 리팩토링 태스크 (순차 추론 필요)

### 2단계: 경량 병렬 (Task run_in_background, 토큰 ~2-3x)

**TeamCreate 없이** Task 도구의 `run_in_background=true`로 병렬 실행.

조건 (모두 충족 시 자동 적용):
- 실행 가능한 독립 태스크가 **3개 이상**
- 모두 **S 또는 M** 복잡도
- 서로 **다른 도메인/파일**을 수정
- 태스크 간 **의존 관계 없음** (blocked_by 없음)

최대 동시 실행: **3개**

### 3단계: 팀 병렬 (TeamCreate, 토큰 ~4-5x)

**사용자가 명시적으로 요청할 때만.** 자동 적용하지 않음.

---

## 자동 판단 플로우

```
1. Glob active/*/pending/*.md 에서 태스크 파일 목록 수집
2. blocked_by 확인 → unblocked 태스크 필터링
3. 도메인/파일 겹침 여부 확인
4. 티어 판단
   ┌─ 독립 태스크 ≤ 2개 → 1단계 (순차)
   ├─ 독립 태스크 ≥ 3개 + 모두 S/M + 파일 겹침 없음 → 2단계 (경량 병렬)
   └─ 사용자 명시 요청 → 3단계 (팀 병렬)
5. 실행
   - 1단계: 순서대로 하나씩
   - 2단계: Task(run_in_background=true) 최대 3개
   - 3단계: TeamCreate → 에이전트 배정
```

## 2단계 실행 패턴

```
1. 각 태스크 파일을 in_progress/로 이동
2. Task(run_in_background=true) 최대 3개 동시 실행
3. TaskOutput으로 결과 수집
4. 완료된 태스크를 completed/로 이동
```

## 도메인 겹침 판단 기준

| 같은 도메인 | 다른 도메인 |
|------------|------------|
| 같은 Entity/Service 수정 | 완전 별개 Entity |
| FK/연관관계 공유 | 별개 API 엔드포인트 |

## 원칙

1. **의심스러우면 순차 처리** — 잘못된 병렬화는 재작업 비용이 더 큼
2. **최대 3개까지만** — 4개 이상은 코디네이션 복잡도가 급증
3. **TeamCreate는 사용자 요청 시에만** — 자동으로 팀을 만들지 않음
4. **결과 검증 후 다음 라운드** — 3개 완료 → 검증 → 다음 3개
