# PLAN 인터뷰 절차

새 로드맵 시작 시, PLAN.md를 작성하기 전에 사용자를 인터뷰하여 요구사항을 구체화한다.

## 역할 제한 (인터뷰 중)

- **사용 가능**: Read, Glob, Grep, AskUserQuestion
- **사용 금지**: Edit, Write, Bash — 인터뷰 중에는 질문만 한다. 구현 약속 금지.
- **응답은 반드시 질문으로 끝낸다** — 정리/요약만 하고 멈추지 않는다.

## 질문 태도

- **아무것도 안 하면?** — "이걸 안 하면 어떻게 되나요?"는 유효한 질문. 문제의 심각도와 우선순위를 드러낸다.
- **더 단순한 접근을 제시한다** — 복잡한 요청에 간단한 대안이 있으면 반문한다.

## 진행 절차

1. **코드베이스 선스캔** — Glob/Read로 프로젝트 구조, 설정 파일, 기존 코드를 먼저 파악한다.
2. **`AskUserQuestion`으로 질문한다** — 선택지를 포함하여 응답 부담을 낮춘다.
   - 브라운필드: 코드에서 발견한 것을 기반으로 확인형 질문 ("X를 발견했는데, Y로 가도 될까요?")
   - 그린필드: 오픈엔드 질문으로 시작
3. **라운드 종료 시 Refine Gate** — 그 라운드의 모든 답변을 한 번에 4필드로 구조화하고 사용자에게 누락/수정 확인을 받는다. (아래 "Refine Gate" 섹션 참조)
4. **최소 3라운드, 모호함이 남아있으면 계속한다.** 사용자가 "됐어"라고 할 때까지.
5. 라운드 사이에 코드베이스 추가 탐색 — 답변에 따라 관련 코드를 읽고, 그 결과로 다음 질문을 구체화
6. 인터뷰 완료 → PLAN.md 초안을 대화에 제시 (각 라운드의 Refine 4필드 결과를 PLAN.md 본문에 반영, 원본은 `### A. 인터뷰 정리` (`## 부록` 아래)에 그대로 박음) → **examiner 에이전트 호출 (필수)**:
   ```
   Agent(
     subagent_type: "examiner",
     description: "PLAN 검토",
     prompt: "다음 PLAN.md 초안의 논리적 허점, 빠진 전제, 실패 시나리오를 검토해줘.\n\n{PLAN.md 초안 전체}"
   )
   ```
7. examiner 보고서를 사용자에게 함께 제시 → AskUserQuestion으로 승인 받기:
   ```
   AskUserQuestion(
     question: "PLAN.md 초안과 examiner 검토 결과를 보고 어떻게 진행할까요?",
     header: "PLAN 승인",
     options: [
       {label: "승인 + 저장", description: "PLAN.md 저장 후 태스크 분해 진행"},
       {label: "수정 필요", description: "추가 인터뷰 라운드"},
       {label: "취소", description: "PLAN 작성 중단"}
     ]
   )
   ```
8. 승인 후 PLAN.md 파일 저장 → 완료 기준을 태스크로 분해 → `active/{로드맵}/pending/`에 생성
9. **승인 없이 태스크 생성 금지. 1회성 작업이면 생략.**

## Refine Gate (라운드 종료 시 1회)

**라운드 정의:** 한 카테고리(아래 "5개 질문 카테고리" 참조)의 1~2개 질문 묶음이 한 라운드다. 사용자가 그 카테고리의 답변을 마쳤거나, 다음 카테고리로 넘어갈 때가 라운드 종료 시점이며 Refine Gate는 정확히 이 시점에 1회 발동.

각 라운드 마지막에 그 라운드의 모든 답변을 4필드로 구조화하고 사용자에게 누락/수정 확인을 받는다.

**목적:** 답변에 묻혀있는 제약(Constraints)·범위 밖(Out of scope)을 드러내, 후속 examiner 검토와 PLAN 저장 시 사용자 의도를 보존한다.

### 4필드 구조

- **Decision**: 이 라운드에서 결정된 내용 (그 라운드의 카테고리별로 한 줄씩)
- **Reasoning**: 사용자가 그렇게 결정한 이유 (답변에 명시됐을 때만; 추측 금지). 비어있으면 "(이유 미진술)"로 표기.
- **Constraints**: 이 결정에 붙는 제약 조건 (환경, 기간, 외부 의존성 등). 없으면 "(없음)".
- **Out of scope**: 사용자가 명시적으로 범위 밖이라 한 항목. 없으면 "(없음)".

### 호출 형식

```
AskUserQuestion(
  question: "이번 라운드 답변을 다음과 같이 정리했어요. 빠진 거 있나요?\n\nDecision: ...\nReasoning: ...\nConstraints: ...\nOut of scope: ...",
  header: "라운드 정리",
  options: [
    {label: "그대로 진행", description: "이 4필드 그대로 PLAN에 반영"},
    {label: "Constraints 추가", description: "제약 조건 추가 입력"},
    {label: "Out of scope 추가", description: "범위 밖 항목 추가 입력"},
    {label: "다시 쓸래", description: "재구성 필요"}
  ]
)
```

> 4필드 본문은 `question`에 넣어 AskUserQuestion description 길이 한계를 피한다. `description`은 짧은 라벨만.

### 규칙

- Refine Gate는 **답변 내부의 누락(Constraints/Out of scope)만** 본다. 전제 검증·실패 시나리오·대안 제시는 examiner 책임이므로 여기서 묻지 않는다.
- "Constraints 추가" / "Out of scope 추가" 선택 시 두 번째 AskUserQuestion으로 누락 텍스트를 받는다.
- "다시 쓸래" 선택 시 4필드를 새로 작성해 다시 확인한다. 무한 루프 방지를 위해 같은 라운드에서 2회까지만 재구성하고, 그래도 사용자가 거부하면 그 라운드의 답변을 원문 그대로 보존하고 다음 라운드로 진행한다.
- Reasoning 필드는 *사용자가 직접 말한 이유만* 적는다. AI가 추론한 이유는 적지 않는다. 사용자가 이유를 말하지 않으면 "(이유 미진술)"로 표기.

### 다층 방어 (examiner와의 관계)

Refine Gate는 *사용자 인지 보조* 1차 검출이다. 사용자가 "그대로 진행"으로 흘려보낸 누락이라도, examiner는 산출물 기반 *흐름 정합성* 관점에서 누락 검토(공통 2)로 2차 검출한다. 두 단계는 시점·정보·검출 방식이 다른 보완 관계 (중첩 아님). examiner의 검출은 *경고 수준* — 차단이 아니므로 사용자 의사를 무력화하지 않는다.

### Refine 결과 보존

라운드 종료 후 4필드 결과는 *대화 내부적으로 누적*하고, 인터뷰 완료 시점에 PLAN.md의 `### A. 인터뷰 정리` (`## 부록` 아래) 섹션에 라운드별로 그대로 박는다. 본문은 4필드를 자연스러운 서술로 반영하되, 부록은 원본 4필드를 유지해 examiner와 후속 출처 마커(`[USER]` 등) 처리의 추적성을 확보한다.

### 부담 평가 (향후 결정)

Refine Gate 의무화로 인한 AskUserQuestion 호출 증가(라운드당 +1회, 보강 시 +2~3회)는 2026-05-12 사용자 합의("써보고 불편하면 그때 바꾸자")에 따라 운영하며 평가한다. 부담 지표(발동 빈도, 사용자 우회율 등)는 향후 결정.

## 5개 질문 카테고리

| 카테고리 | 핵심 질문 | 사용 시점 |
|----------|----------|----------|
| ① 본질 | "이것의 본질은 뭔가요? 표면적 요청과 실제 목표가 같은가요?" | **항상 첫 번째** |
| ② 근본 원인 | "이게 근본 원인인가요, 증상인가요? 이걸 해결하면 진짜 문제가 풀리나요?" | 버그/개선 요청 시 |
| ③ 숨겨진 가정 | "당연하다고 전제하는 것이 있나요? 반대가 사실이라면?" | 접근 방식이 이미 정해져 있을 때 |
| ④ 기존 맥락 | "현재 코드베이스에서 관련된 부분은? 건드리면 안 되는 것은?" | 브라운필드 프로젝트 |
| ⑤ 완료 조건 | "어떻게 되면 '완료'인가요? 구체적으로." | **항상 마지막** |

적응적 규칙: 본질로 시작, 답변이 구체적이면 완료 조건으로 건너뜀. 라운드당 질문 1~2개.

## 이미 계획이 있는 경우

사용자가 이미 작성한 계획이나 플랜 모드 결과가 있어도 태스크 분해는 건너뛰지 않는다. 계획을 PLAN.md로 옮긴 뒤 정상 절차 진행.
