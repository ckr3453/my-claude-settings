# PLAN 인터뷰 절차

새 로드맵 시작 시, PLAN.md를 작성하기 전에 사용자를 인터뷰하여 요구사항을 구체화한다.

## 역할 제한 (인터뷰 중)

- **사용 가능**: Read, Glob, Grep, AskUserQuestion
- **사용 금지**: Edit, Write, Bash — 인터뷰 중에는 질문만 한다. 구현 약속 금지.
- **응답은 반드시 질문으로 끝낸다** — 정리/요약만 하고 멈추지 않는다.

## 질문 태도

- **증상이 아닌 근본 원인을 찾는다** — "X 기능 추가해줘"에 "왜 X가 필요한가?"를 먼저 묻는다. 해결하려는 실제 문제가 다를 수 있다.
- **가정을 뒤집는다** — 사용자가 당연히 전제하는 것을 명시적으로 질문한다. "DB가 필요하다" → "정말 DB가 필요한가? 파일로 충분하지 않은가?"
- **아무것도 안 하면?** — "이걸 안 하면 어떻게 되나요?"는 유효한 질문. 문제의 심각도와 우선순위를 드러낸다.
- **더 단순한 접근을 제시한다** — 복잡한 요청에 간단한 대안이 있으면 반문한다.

## 진행 절차

1. **코드베이스 선스캔** — Glob/Read로 프로젝트 구조, 설정 파일, 기존 코드를 먼저 파악한다.
2. **`AskUserQuestion`으로 질문한다** — 선택지를 포함하여 응답 부담을 낮춘다.
   - 브라운필드: 코드에서 발견한 것을 기반으로 확인형 질문 ("X를 발견했는데, Y로 가도 될까요?")
   - 그린필드: 오픈엔드 질문으로 시작
3. **최소 3라운드, 모호함이 남아있으면 계속한다.** 사용자가 "됐어"라고 할 때까지.
4. 라운드 사이에 코드베이스 추가 탐색 — 답변에 따라 관련 코드를 읽고, 그 결과로 다음 질문을 구체화
5. 인터뷰 완료 → PLAN.md 초안을 대화에 제시 → **`examiner` 에이전트를 반드시 호출하여 논리적 허점/빠진 전제/실패 시나리오를 검토** → examiner 보고서를 사용자에게 함께 제시 → 사용자 승인 대기
6. 승인 후 PLAN.md 파일 저장 → AC를 태스크로 분해 → `active/{로드맵}/pending/`에 생성
7. **승인 없이 태스크 생성 금지. 1회성 작업이면 생략.**

```
코드베이스 선스캔 → AskUserQuestion 인터뷰 (모호함 해소까지) → PLAN.md 초안 제시 → 승인 → 파일 저장 + 태스크 분해
```

## 5개 질문 카테고리

| 카테고리 | 핵심 질문 | 사용 시점 |
|----------|----------|----------|
| ① 본질 | "이것의 본질은 뭔가요? 표면적 요청과 실제 목표가 같은가요?" | **항상 첫 번째** |
| ② 근본 원인 | "이게 근본 원인인가요, 증상인가요? 이걸 해결하면 진짜 문제가 풀리나요?" | 버그/개선 요청 시 |
| ③ 숨겨진 가정 | "당연하다고 전제하는 것이 있나요? 반대가 사실이라면?" | 접근 방식이 이미 정해져 있을 때 |
| ④ 기존 맥락 | "현재 코드베이스에서 관련된 부분은? 건드리면 안 되는 것은?" | 브라운필드 프로젝트 |
| ⑤ 완료 조건 | "어떻게 되면 '완료'인가요? 구체적으로." | **항상 마지막** |

적응적 규칙: 본질로 시작, 답변이 구체적이면 완료 조건으로 건너뜀. 라운드당 질문 1~2개.

## 이미 계획이 있는 경우

사용자가 이미 작성한 계획이나 플랜 모드에서 나온 계획이 있을 때:

1. **반드시 태스크 파일로 분해한 후 실행한다** — 계획이 있다고 태스크 분해를 건너뛰지 않는다
2. 계획을 PLAN.md로 옮기고 → 태스크 분해 → pending/에 생성
3. 이후 정상 워크플로우 진행 (태스크 시작 → 구현 → 검증 → 완료)
